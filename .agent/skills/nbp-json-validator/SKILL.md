# NBP JSON Prompt Validator

## Цель (Goal)
Проанализировать код сайта (или предоставленный тестовый вывод), отвечающий за сборку опций пользователя в итоговый JSON-запрос для модели Nano Banana Pro. Убедиться, что генерируемый JSON строго соответствует эталонной структуре, не содержит конфликтующих данных и синтаксических ошибок.

## Инструкции (Instructions)

1. **Изучение эталона (Reference Check):**
   - Открой и изучи эталонный файл `examples/reference_nbp.json`. Это идеальная структура, которую должен генерировать сайт пользователя.
   - Обрати внимание на типы данных (строки, булевы значения, вложенные объекты).

2. **Анализ сборки JSON (Payload Generation Analysis):**
   - Проанализируй функции сайта (JS/TypeScript/Python), которые собирают состояния UI (выбранные стили, соотношение сторон, негативные промпты) в итоговый JSON-объект.
   - Проверь, правильно ли экранируются специальные символы и кавычки (особенно важно для текстовых промптов NBP).

3. **Проверка логической целостности (Logic & Conflict Validation):**
   - Проверь отсутствие пустых или `null` значений там, где они обязательны (например, поле `prompt`).
   - Убедись, что взаимоисключающие параметры не попадают в JSON одновременно (например, если пользователь выбрал `aspect_ratio: "1:1"`, в JSON не должны улетать остаточные данные от прошлых выборов, вроде `16:9`).
   - Если сайт использует функции аннотирования или обводки (editing annotations), проверь, корректно ли их координаты или маски передаются в JSON.

4. **Генерация отчета (Output):**
   - Если код сборки генерирует невалидный JSON, покажи, где именно происходит сбой (например, ошибка конкатенации строк).
   - Выведи сравнение: "То, что генерирует сайт" vs "То, как требует эталон".
   - Напиши исправленный код функции-сборщика (builder function), который гарантированно возвращает структуру, идентичную `examples/reference_nbp.json`.

## Ограничения (Constraints)
- Не меняй структуру эталонного JSON. Сайт должен подстраиваться под API, а не наоборот.
- При проверке строгих типов (например, если API ждет `boolean` для `text_accuracy_mode`, а сайт отправляет строку `"true"`), обязательно помечай это как критическую ошибку.
